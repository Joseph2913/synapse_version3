## What this stage owns

Writing deduplicated entities to `knowledge_nodes`, writing relationships to `knowledge_edges`, computing and storing node and edge embeddings, writing transcript chunks, and triggering cross-connection discovery. This stage runs after Stage 6 dedup and produces the final graph state.

---

## Architecture

### Before Block 2 (eliminated)

`src/services/extractionPersistence.ts` exported `saveNodes`, `saveEdges`, `updateNodeEmbeddings`, `updateEdgeEmbeddings` for use by the browser-side pipeline. These ran directly from the browser, which required the Supabase anon key to write to `knowledge_nodes` and `knowledge_edges` (RLS-gated by `user_id`). Edge embeddings were a separate post-save step. Duplicate edges were not constrained at the DB level.

### After Block 2 (current)

All persistence runs inside `api/ingest/extract-persist` (interactive path) or `api/pipeline/extract-pipeline.ts:runExtractionCore()` (headless path). The browser sends reviewed entities to `extract-persist` via `POST` and receives a summary response. No graph-write code remains in `src/`.

## Node embeddings

Embedding text: `${label}\n\n${description}` (confirmed rule ŌĆö same as before Block 2). Generated by `batchEmbed` in Stage 7, stored in `knowledge_nodes.embedding` (vector(3072)).

---

## Edge embeddings

Embedding text: `${source_label} ŌåÆ ${relation_type} ŌåÆ ${target_label}`. Computed inline in `runExtractionCore()` immediately after `saveEdges`:

1. Fetch saved edge IDs and node IDs from the DB.
2. Build embedding texts by joining node labels via the relation type.
3. Call `batchEmbed` on all edge texts.
4. Bulk-update `knowledge_edges.embedding` in one upsert.

**Failure policy:** non-blocking. If edge embedding fails, `logError` is called with `stage:'knowledge', status:'skipped'` and the pipeline continues. Source status is not affected.

---

## Edge dedup

Two layers:

**Database constraint:** `knowledge_edges_dedup_uniq` ŌĆö UNIQUE INDEX on `(user_id, source_node_id, target_node_id, relation_type)`. Migration: `supabase/migrations/20260428_edge_dedup_constraint.sql`. The migration first deduplicates existing duplicate edge groups (keeps the row with highest `weight`, breaking ties by earliest `id`) before creating the index.

**Application layer:** `saveEdges` uses `.upsert()` with `onConflict: 'user_id,source_node_id,target_node_id,relation_type', ignoreDuplicates: true`. Duplicate edge writes from the same or concurrent extractions are silently discarded.
