# Simulate Feature — Synapse × MiroFish Integration

**Status:** Implemented and operational (fallback path active, OASIS pending Python 3.12 upgrade)  
**Repos:** `synapse_version3` (Synapse UI) + `synapse-simulate-sidecar` (Python backend)  
**PRDs:** PRD-Simulate-A (Synapse side) · PRD-Simulate-B (Sidecar side)

---

## What This Is

The Simulate feature turns Synapse's knowledge graph into a **prediction engine**. The user selects a topic scope from their existing graph, writes a natural-language prediction question, and receives a structured forecast — generated by running a multi-agent social simulation over their curated knowledge.

The simulation engine is **MiroFish**, an open-source Python framework built on CAMEL-AI's OASIS engine. It spawns AI agents with distinct personalities derived from the graph, runs them through simulated social interaction, and surfaces emergent predictions. From the user's perspective, Simulate is a native Synapse feature. MiroFish is entirely abstracted.

## Setup Process

### What Was Built

Two separate repositories with a clean HTTP boundary between them:

**`synapse_version3` (Synapse side)**
- New `/simulate` route and `SimulateView` page
- `ScopeSelector`, `PredictionInput`, `PersonaPreview`, `SimulationRunner`, `SimulationReport` components
- `useSimulate` and `useSimulationJobs` hooks
- `src/services/simulate.ts` — graph export, sidecar trigger, Supabase writes
- `simulation_jobs` Supabase table with RLS
- `VITE_SIMULATE_SIDECAR_URL` env variable (defaults to `http://localhost:8000`)

**`synapse-simulate-sidecar` (Python side)**
- FastAPI server (`main.py`) with `/health` and `/simulate` endpoints
- MiroFish engine vendored from `MiroFish-Offline` fork into `mirofish/app/`
- Five pipeline modules: `graph_import`, `persona_gen`, `environment_setup`, `simulation_runner`, `report_gen`
- Supabase writer for async result persistence

### Roadblocks and Resolutions

**1. MiroFish uses Flask + Neo4j; sidecar uses FastAPI + Supabase**  
The vendored MiroFish code had eager imports pulling in Flask blueprints, Neo4j drivers, and `transformers`. These were replaced with lazy-import stubs and a no-op `__init__.py`. The simulation engine itself was kept intact.

**2. `camel-ai` and `oasis-sim` require Python 3.10+; machine had Python 3.9**  
Packages were removed from `requirements.txt`. OASIS now runs as a subprocess spawned by the sidecar. A fallback path was implemented: when the OASIS subprocess fails, Gemini synthesises a report directly from agent profiles. Full OASIS runs require Python 3.12 (pending installation).

**3. Supabase service key in `sb_secret_` format rejected by `supabase-py` v2**  
The key format was incorrect. Replaced with the JWT service role key from Supabase dashboard → Project Settings → API.

**4. Supabase progress writes deadlocking the async pipeline**  
Calling synchronous Supabase HTTP writes inside FastAPI's async event loop caused the pipeline to hang after the OASIS stage. All intermediate `update_job_status` calls were replaced with a fire-and-forget threading pattern (`daemon=True` threads) that never block the pipeline.

**5. Edge query exceeding browser URL length limit**  
The `buildSeedGraph` function was passing hundreds of node IDs into a PostgREST `.in()` filter, generating URLs over 100KB. Browser rejected them with `ERR_FAILED`. Fixed by: (a) capping nodes at 150 sorted by centrality, (b) replacing the edge query with a Supabase RPC function `get_scoped_edges` that runs server-side, (c) fetching source chunks by time window rather than node ID list.

**6. Pydantic 422 validation mismatch between Synapse payload and sidecar model**  
The field names sent by the Synapse frontend didn't exactly match the Pydantic model on the sidecar. Resolved by adding request body logging to identify the mismatch and aligning field names.

**7. Python 3.14 incompatible with pinned dependencies**  
`pydantic-core==2.18.1` has no wheel for Python 3.14. Downgraded target to Python 3.12 (pending manual installation from python.org).

## Architecture

```
Browser (Synapse UI)
    │
    ├─ GET /health          → confirms sidecar is running (on page load)
    │
    └─ POST /simulate       → sends seed graph JSON + prediction question
           │
    FastAPI Sidecar (localhost:8000)
           │
           ├─ Returns {"status": "accepted"} immediately
           │
           └─ Background task runs pipeline:
                  │
                  ├── graph_import.py       Synapse JSON → internal graph
                  ├── persona_gen.py        Nodes → agent personas (LLM)
                  ├── environment_setup.py  World context + what-if injection
                  ├── simulation_runner.py  OASIS subprocess (or fallback)
                  ├── report_gen.py         LLM synthesis → structured JSON
                  └── supabase_writer.py    Writes result to simulation_jobs
                                                    │
Browser (Synapse UI) ←── Supabase realtime ─────────┘
    polls simulation_jobs for status updates
```

## What Happens When You Click Run

### In the UI

1. **Scope selector** — user picks anchor nodes and a time window (30 days / 90 days / 6 months / all time). A scope preview shows entity, edge, and source counts.
2. **Prediction input** — user writes a free-text prediction question. Optional what-if variables can be injected (e.g. "Assume GPT-5 launches in Q2").
3. **Persona preview** — Person and Organization nodes from the selected scope are shown as read-only auto-generated agents. Users can exclude individual agents but cannot create or edit them.
4. **Run** — button is disabled until: sidecar is healthy, at least one anchor is selected, and a question is entered.
5. **Runner view** — progress bar with stage labels. The page can be navigated away from; a pulsing badge on the Simulate nav item indicates an active run.
6. **Report view** — structured prediction report renders when `status = completed` in Supabase.

### In the Background

**Synapse (frontend):**
1. Calls `buildSeedGraph()` — queries Supabase for nodes, edges, and source chunks within the time window. Caps at 150 nodes by centrality. Uses the `get_scoped_edges` RPC for edge fetching.
2. Creates a `simulation_jobs` row in Supabase with `status: pending`.
3. POSTs the seed graph JSON + question + what-if variables to `localhost:8000/simulate`.
4. Begins polling Supabase via realtime subscription for status changes.

**Sidecar (Python):**
1. Returns `{"status": "accepted"}` immediately — never holds the HTTP connection open.
2. Starts a background task via FastAPI's `BackgroundTasks`.

## MiroFish Pipeline — Stage by Stage

### Stage 1 — Graph Import (`graph_import.py`)
**Logic owner: Synapse** (provides the data) / **executed by: sidecar**

Takes Synapse's serialised subgraph JSON and converts it into MiroFish's internal `InternalGraph` structure. Builds adjacency maps and per-node edge indexes for fast traversal. No LLM calls. Pure data transformation.

Input: `SeedGraph` (nodes, edges, source chunks, metadata)  
Output: `InternalGraph` with adjacency maps

### Stage 2 — Persona Generation (`persona_gen.py`)
**Logic owner: MiroFish pattern + sidecar adaptation**

Filters the graph for `Person`, `Organization`, and `Team` nodes. Sorts them — anchor nodes first, then by centrality (edge count). Caps at 30 agents (`MAX_AGENTS` env var).

For each eligible node, makes one LLM call to Gemini to write a **behavioural persona prompt** — a 3–5 sentence description of how this entity thinks, what motivates it, and how it acts in context of the prediction question. This prompt is later passed to the OASIS simulation engine to govern agent behaviour.

Influence tier is derived automatically:
- `high` → anchor node OR centrality ≥ 5
- `medium` → centrality ≥ 2
- `low` → everything else

Input: `InternalGraph` + prediction question  
Output: List of `SimulationAgent` objects with personality prompts

### Stage 3 — Environment Setup (`environment_setup.py`)
**Logic owner: sidecar**

Assembles the "world" the agents will inhabit. Samples up to 3 source chunks per source (capped at ~32,000 characters total) from Synapse's `source_chunks` — these are the 500-token passages from ingested YouTube videos, documents, and transcripts. This raw knowledge becomes the simulation's ground truth.

What-if variables are injected here as environmental conditions — they become facts the agents operate within.

The final `world_context` string combines: prediction question + what-if conditions + key entity summary + sampled source content.

Input: Agents, seed graph, prediction question, what-if variables  
Output: `SimulationEnvironment` with assembled world context

### Stage 4 — Simulation Runner (`simulation_runner.py`)
**Logic owner: MiroFish (OASIS engine)**

**With OASIS (Python 3.12 + camel-ai — pending):**  
The `SimulationManager` creates a simulation directory. `OasisProfileGenerator` converts agent personas into OASIS-format CSV/JSON profiles. `SimulationConfigGenerator` makes a 4-step LLM call to generate time configuration, event configuration, agent activity parameters, and platform configuration. The OASIS subprocess is then spawned and polled every 5 seconds. Agents interact across two simulated platforms (Twitter-like broadcast + Reddit-like threaded) over multiple rounds, posting, responding, shifting opinions, and influencing each other.

**Fallback path (current — Python 3.9):**  
When the OASIS subprocess fails, the pipeline detects the failure and skips to report generation using agent profiles and world context directly. The report quality is lower (no emergent interaction data) but structurally valid.

Input: `InternalGraph`, `SimulationEnvironment`, `List[SimulationAgent]`  
Output: `SimulationResult` (agent states, interaction log, emergent signals)

### Stage 5 — Report Generation (`report_gen.py`)
**Logic owner: sidecar (Gemini as ReportAgent)**

A single LLM call to Gemini 2.0 Flash acting as a **prediction analyst**. The agent has access to:
- The original prediction question
- The injected what-if conditions
- A summary of agent interactions from the simulation (last 50 interactions)
- Final agent states (each agent's position at end of simulation)
- Emergent signals detected by OASIS
- (Fallback: agent persona prompts and world context directly)

The agent is instructed to respond **only in structured JSON** matching a fixed schema. Temperature is set to 0.3 for consistency.

**Output schema:**
```json
{
  "headline": "One-sentence core prediction",
  "summary": "2-3 sentence overview",
  "forecasts": [
    {
      "direction": "What will happen",
      "rationale": "Why",
      "timeframe": "When",
      "confidence": "low | medium | high"
    }
  ],
  "agent_moves": [
    {
      "agent_label": "Entity name",
      "entity_type": "Organization",
      "likely_action": "What they will do",
      "rationale": "Why",
      "influence": "low | medium | high"
    }
  ],
  "surprises": ["Unexpected emergent behaviours"],
  "confidence_level": "low | medium | high",
  "confidence_rationale": "Why this confidence level"
}
```

If JSON parsing fails, a fallback minimal report is returned — the simulation is not marked as failed.

### Stage 6 — Supabase Write (`supabase_writer.py`)
**Logic owner: sidecar**

Writes the completed `SimulationReport` to `simulation_jobs.result` (JSONB) and sets `status = completed`. Uses the Supabase service role JWT key to bypass RLS. The Synapse frontend's realtime subscription detects the status change and transitions the UI from the runner view to the report view.

## The Report Agent

The report is generated by a **single Gemini 2.0 Flash call** with a fixed system prompt instructing it to act as a prediction analyst. It is not a persistent agent — it is a one-shot synthesis call.

**What it has access to:**
- The prediction question and what-if conditions
- Last 50 agent interactions from the simulation log
- Final state of each agent (their position/stance at end of simulation)
- Emergent patterns detected by OASIS
- In fallback mode: agent personality prompts + full world context (source chunks)

**What it does not have access to:**
- The full raw graph (only what was included in the world context)
- External web search or real-time data
- Previous simulations or historical context

**How it presents the output:**
Strictly JSON. No prose preamble. The structured output is then mapped to typed `SimulationReport` and `SimulationForecast` / `SimulationAgentMove` objects in Python, and stored as JSONB in Supabase. The Synapse frontend renders each field as a distinct UI component — headline, forecast cards, agent move list, surprise signals, confidence badge.
